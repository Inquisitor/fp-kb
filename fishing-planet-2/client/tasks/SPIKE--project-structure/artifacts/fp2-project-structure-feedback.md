# FP2 Client Project Structure Design — фідбек

_Дата:_ 2026-02-16  
_Контекст:_ аналіз структури клієнта FP2 (Unity 6 HDRP) на основі документа “Fishing Planet 2 — Client Project Structure Design” (2026-02-09).

---

## 1) Що вже зроблено сильно

### Репозиторії та ролі
- **fp2-client / fp2-art / fp2-packages**: здоровий поділ відповідальності.
- Роль **Tech Artist як «шлюз»** між art і client з WIP → Review → Export зменшує ризик зіпсувати main-проєкт та скорочує шум у Merge Requests.

### Структура Assets (гібрид: модулі + shared)
- Модулі верхнього рівня (**Fishing/Environment/Character/Economy/UI/Audio**) плюс **`_Core`** читаються швидко, масштабуються без хаосу.
- Єдине **`Scenes/`** для всіх сцен полегшує навігацію та build-процес, а також дисциплінує команду.

### Assembly Definitions + правила залежностей
- Чітко окреслено:
  - **`FP2.Core` без залежностей**,
  - модулі залежать **лише від Core**,
  - міжмодульна комунікація через **SignalBus**,
  - `.Editor` асемблії окремо.
- Це підвищує швидкість компіляції та прибирає «сповзання» залежностей у майбутньому.

### Архітектура (Zenject + Signals + FSM)
- **ProjectContext у Boot** як єдина точка входу: стабільний lifecycle.
- **Hierarchical FSM** добре підходить під домен гри (глобальні стани + вкладені стани підсистем).

### Сценовий флоу (Single + Additive)
- Boot як Scene 0 + Additive сцени (HUD/Weather/Audio) реалістично зменшують конфлікти між рівнями/UI/аудіо.

### CI/CD і тестування
- Модель Validate (MR), develop-build, release-build, nightly виглядає практично.
- Окремо важливо: **Asset Validation** тестами (неймінг, LODGroup, missing scripts) — це правильна страховка від людських помилок.

## 2) Місця ризику / «де може заболіти»

### A) Git LFS + UnityYAMLMerge: потенційна нестикування політики
У `.gitattributes` Unity-файли (`*.unity`, `*.prefab`, `*.asset`, `*.controller`) налаштовані як `merge=lfs`. Це означає, що **фактично використовується LFS merge driver**, і заявлена стратегія "UnityYAMLMerge для сцен" може не працювати так, як очікується.

**Чому це критично:** сцени/префаби — головне джерело конфліктів у команді 15–30 людей.

**Рекомендовані напрямки (оберіть один як “single source of truth”):**
1. **Merge-friendly (UnityYAMLMerge як основний шлях):** не трекати `.unity/.prefab/.asset` через LFS (лишити YAML текстом), використовувати UnityYAMLMerge + нормальні diff’и.
2. **Lock-first (LFS locking як основний шлях):** лишити LFS для `.unity/.prefab`, але **чесно** оформити правило: “сцени мерджимо рідко, працюємо локами; конфлікти розрулює відповідальний (TA/Lead)”. Тоді UnityYAMLMerge описати як “manual emergency path”, а не як дефолт.
3. **Просунутий варіант:** кастомний merge driver, який поєднує LFS і UnityYAMLMerge (потрібна підтримка й дисципліна).

> Моя порада: або **(1)** якщо ви реально хочете мерджити сцени, або **(2)** якщо ставка на locking і мінімум конфліктів.

### B) `UI/Resources` як зона ризику по пам’яті й контролю завантаження
`Resources/` часто еволюціонує у “склад без інвентаризації”. Якщо там плануються атласи/шрифти/важкі UI-асети, з часом складно контролювати:
- що завантажується,
- коли завантажується,
- чому з’їдає пам’ять.

**Рішення:** або жорстке правило “Resources тільки для bootstrap”, або перехід на **Addressables** для UI контенту.

### C) Правило “UI depends on modules (readonly)” може непомітно перетворитись на сильну зв’язність
На практиці UI починає тягнути внутрішні сервіси й моделі модулів. Потім рефакторинг болить.

**Контракт, який варто закріпити:**
- UI читає або **ReadModels/DTO**, або працює через **фасади інтерфейсів** чи **signals**.
- Винести “контракти презентації” в окремий шар (`FP2.PresentationContracts` або в `FP2.Core.Data`), щоб UI не чіпляв “нутрощі” модулів.

### D) Signals можуть перетворитись на “event soup”
SignalBus чудовий, але без гігієни легко отримати сотні сигналів без зрозумілої структури.

**Мінімальна гігієна:**
- cross-module сигнали живуть у **Core/Events** (або окремій збірці Events),
- внутрішні сигнали — в межах модуля,
- заборона на “per-frame signals”,
- неймспейси/папки за доменом: `FP2.Core.Events.Fishing`, `...Economy`, `...UI`.

### E) `.unitypackage` в art pipeline: ризики GUID та рев’ю змін
Unitypackage зручний, але може ускладнювати рев’ю й контроль посилань (GUID/мета-файли).

**Запобіжники:**
- експорт тільки з контрольованими `.meta`,
- автоматична валідація після імпорту в client,
- у перспективі — міграція зрілих наборів контенту в **UPM packages** (як ви й закладаєте в `fp2-packages`).

## 3) Рекомендації, які дадуть найбільший ефект (пріоритезовано)

### Пріоритет 1: узгодити “LFS locking vs UnityYAMLMerge” і зафіксувати політику
Додайте в дизайн 1 сторінку “правда-таблицю”:
- які файли **lockable**,
- які файли **реально мерджимо**,
- який merge driver використовується для `.unity/.prefab`,
- що робимо при конфлікті (кроки + відповідальний).

Це прибере левову частку майбутніх проблем.

### Пріоритет 2: контракт для Prefabs / Art, щоб не було дублювання “істини”
Зараз є `Assets/Art/**` і `*/Prefabs/`. Варто чітко розвести призначення:
- `Assets/Art/` — render-асети (mesh, textures, матеріали, анімації, view-prefabs без логіки),
- `Assets/{Module}/Prefabs/` — gameplay prefabs (скрипти, колайдери, ін’єкція, інтеракції),
- (опційно) суфікси `_View`, `_Gameplay` або правило неймінгу.

### Пріоритет 3: anti-chaos правила для Signals
Додайте короткий гайд:
- коли сигнал vs прямий виклик,
- структура сигналів (папки/неймспейси),
- правила життєвого циклу підписок (де гарантуємо unsubscribe).

### Пріоритет 4: policy для `Resources/`
Одна жорстка фраза (і перевірка в рев’ю/валідації):
- або “Resources лише для bootstrap”,
- або “Resources дозволено тільки для X/Y” (конкретно).

## 4) Нотатки по FSM (щоб не вилізли «примари» в runtime)

Ваш `StateMachine<TState>` з `Resolve<T>()` на `Enter` — робочий патерн. Я б додав явні правила:
- **Transient vs Cached** стани: які створюються щоразу, а які живуть довго.
- Єдиний стиль керування підписками: або `Subscribe`/`Unsubscribe` в `Enter/Exit`, або гарантії lifecycle від Zenject (і тоді це теж задокументувати).
- Хто викликає `Tick()` (ITickable чи апдейтер) і порядок тіку для вкладених FSM.

## 5) Підсумок

Каркас проєкту виглядає зрілим: модулі, залежності, сцени, Zenject/FSM, CI та валідації. Найбільший “ножик у темряві”, який варто прибрати вже зараз, — це **узгодження реальної стратегії для сцен/префабів: LFS locking проти UnityYAMLMerge**. Якщо це прояснити й зафіксувати як політику, проєкт буде набагато менш конфліктним для команди ~20 людей.
