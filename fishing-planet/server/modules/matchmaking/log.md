# Decision Log — Matchmaking

## 2026-04-29
- Incident: rating-related competition feature (`Places[i].Rating`, `NoShowRatingPenalty`, `ZeroScoreRatingPenalty`, `Grouping`) shipped to Steam prod today. 7 competition Tournament rows scheduled for today/tomorrow (IDs 318866–318872) were generated by the **previous** binary and so missed all four fields in their `ConfigJson`. Standard remediation `RegenerateFutureCompetitions` skipped them because their `RegistrationStart` was already in the past (registrations existed: 506 rows / 268 distinct users). `RegenerateCompetitionsRandom` would have regenerated them but at the cost of all those registrations
- Hot-patched in place via point `JSON_MODIFY` (idempotent, values copied from each Tournament's Template):
  - All 7 (live included): `Places[i].Rating` 1..10 and root-level `NoShowRatingPenalty` / `ZeroScoreRatingPenalty`
  - 6 upcoming (live excluded): `Grouping` (whole object). Skipped on live because `ProcessGrouping` runs at tournament start — that moment was already past for live, no `BracketId`s to assign retroactively. All 6 upcoming will group correctly on their `StartDate`. Live 318866 plays as one common pool — acceptable since `RatingMultiplier=1.0` means rating math is identical, only bracket-level ranking is missing
  - Places > 10 not added — by design (penalties flow through `*RatingPenalty` now, not legacy tail places)
- `past` cohort (713) is history, not a defect — they were generated AND played under the pre-feature binary, on the contract they actually had. Nothing to patch. `far-future` cohort (142) generated after deploy, already healthy
- Lesson: any `TournamentJsonConfig` schema extension creates this exact gap — future competitions generated before deploy will be missing the new fields, and any of them with active registrations cannot be regenerated. Release procedure for such changes should include a `JSON_MODIFY`-based backfill step targeting `Tournaments` rows with `EndDate > now` whose Template defines the new fields. Without it, the only options are losing user state (cascade-delete via `RegenerateCompetitionsRandom`) or hand-patching, as we did here
- Finding (evening): the morning `Grouping` backfill via `JSON_QUERY(tt.ConfigJson, '$.Grouping')` + `JSON_MODIFY` was incomplete. Templates never store `Brackets[i].MaxRating` — it is auto-computed by `MatchmakingLogic.InitializeGrouping` during template deserialization. The two `TournamentsHelper.FromDto` overloads are asymmetric: `FromDto(TournamentTemplateDto)` calls `InitializeGrouping`, but `FromDto(TournamentDto)` (used by `TournamentStartAdapter` → `ProcessGrouping`) does not. The regular generation pipeline sidesteps this by deserializing the template first (initializes `Grouping`), then re-serializing the now-complete object into the Tournament's `ConfigJson`. Raw-template `JSON_MODIFY` copy bypasses the initializer, so `int MaxRating = 0` per bracket → `MatchmakingLogic.CreateBuckets` filters by inverted ranges and admits only `CompetitionRating == 0` participants into bracket 1 (the rest end up unassigned). Confluence Matchmaking spec (page 5505613835) confirms `MaxRating` is auto-computed and "do not set manually" — formalising the implicit contract this defect class breaks
- Mitigated: per-bracket `MaxRating` (100 / 1000 / 2147483647 — recomputed via `MinRating[i+1] - 1` for non-last brackets, `int.MaxValue` for the last) backfilled into the two still-upcoming competitions (318871, 318872) via point `JSON_MODIFY`, applied >5 minutes before each `StartDate`. The three already-grouped tournaments (318868 ended, 318869 ended, 318870 live) left as-is — `ProcessGrouping` already executed and `RatingMultiplier = 1.0` makes per-bracket rating math equivalent
- Lesson: code-side defensive fix is the right place for this class — invoke `MatchmakingLogic.InitializeGrouping` in `MatchmakingLogic.ProcessGroupingForTournament` (local) or symmetrically in `TournamentsHelper.FromDto(TournamentDto)` (closes the gap for any future caller, mirrors the existing `FromDto(TournamentTemplateDto)` overload). Either turns the "MaxRating must be persisted in `ConfigJson`" implicit contract into a self-healing one — future template-`Grouping` `JSON_MODIFY` backfills no longer need a parallel `MaxRating` step. Strongly recommended to land before next platform deploy

## 2026-03-09
- DB rename approach: full `sp_rename` + ConfigJson `REPLACE` instead of DAL mapper abstraction. Simpler, no runtime overhead, atomic with downtime
- ConfigJson dead param removal: `REPLACE` (preserves formatting) over `JSON_MODIFY` (reformats entire JSON). Game designers rely on specific indentation/line breaks
- Patch idempotency: `IF EXISTS` for DDL, `REPLACE` with `WHERE LIKE` for data ops — safe for re-execution after version bump
- Lesson: handle both CRLF and LF in ConfigJson REPLACE patterns (LF fallback pass)
