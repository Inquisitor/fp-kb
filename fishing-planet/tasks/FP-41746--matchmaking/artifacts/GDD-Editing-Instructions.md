## Правка 1 — Глоссарий: переписать определения и добавить NSR, WSR, FFS

**Где:** секция `## Glossary`, все определения от **Bracket** до **Group** включительно

**Что сделать:** заменить определения Bracket, Bucket, Group на уточнённые (с примерами в компактной нотации), и добавить три новых термина (NSR, WSR, FFS) после Group.

**Почему:**
- **Bracket** — уточнено: это настройка из JSON, именованный диапазон рейтинга, а не "розподіл гравців". Убрана нотация `_(10 / 6 / 7)_`, не относящаяся к брекетам.
- **Bucket** — переписан: рабочий контейнер, 1:1 с брекетами на старте, балансируется. Введена компактная нотация `[42 / 23 / 31]` с примерами.
- **Group** — дополнен: примеры разного распределения трёх групп по бакетам, пояснение про тип винагороди.
- **NSR, WSR, FFS** — новые термины, реализованные в коде.

**Готовый блок для замены (от `**Bracket**` до конца глоссария):**

```markdown
**Bracket** (брекет)
Іменований діапазон змагального рейтингу (PCR), визначений у JSON змагання. Використовується для початкової класифікації гравців. Діапазони брекетів не повинні перетинатися або мати прогалини.

Приклад: нуби (0–100), мідли (101–1000), топи (1001+) — три брекети.

---

**Bucket** (бакет)
Робочий контейнер гравців під час матчмейкінгу. На початку алгоритму для кожного брекета створюється рівно один бакет, і кожен гравець потрапляє до бакета, що відповідає його брекету. Під час балансування гравці можуть переміщуватися між бакетами, щоб кожен мав щонайменше `MinSize` учасників.

Приклад (`MinSize` = 20, 96 учасників): 42 гравці з брекета «Нуби», 23 з «Мідли» та 31 з «Топи» — початкові бакети (від найслабшого до найсильнішого): `[42 / 23 / 31]`.

Якщо з певного брекета не прийшло жодного учасника, відповідний бакет залишається порожнім протягом усього матчмейкінгу: `[42 / 0 / 54]`.

---

**Group** (група)
Фінальний результат матчмейкінгу. Набір гравців, які змагаються між собою та розподіляють повний комплект винагород. Кожен непорожній бакет поділяється на одну або більше груп.

Наприклад, три групи можуть бути розподілені по-різному: по одній у кожному бакеті `[42 / 23 / 31]`, дві в одному бакеті `[42 / 0 / 27+27]`, або всі три в одному `[32+32+32 / 0 / 0]`.

Тип винагороди групи визначається за брекетом, до якого належить більшість її учасників. Наприклад, якщо в групі переважають нуби — винагорода буде нубівською. Якщо до 1 нуба приєднали 19 топів — винагорода буде як у топів.

---

**Правило заборони поділу (No-Split Rule, NSR)**
Якщо до найслабшого бакета (MinRating = 0) були переміщені гравці з сильніших брекетів під час балансування — цей бакет **не підлягає поділу на декілька груп** за жодних умов. NSR має найвищий пріоритет і діє навіть якщо MaxSize, MaxGroupCount або TargetSize вимагають поділу.

---

**Правило "слабшим — менші групи" (Weak-Small Rule, WSR)**
При розподілі гравців на групи, слабші бакети отримують пріоритет на створення більшої кількості менших груп. Залишки (якщо кількість гравців не ділиться порівну) розподіляються до сильніших груп.

---

**Стратегія справедливого розподілу (Fair Fill Strategy, FFS)**
Алгоритм розподілу бюджету груп при використанні MaxGroupCount. Оптимізує кількість груп у кожному бакеті так, щоб розмір груп був якомога ближчим до TargetSize (якщо вказано) або щоб бюджет було освоєно максимально ефективно. Детальний опис — див. TDD _(TODO: додати посилання)_.
```

## Правка 9 — убрать "Мідли як наповнювач"

**Где:** два места в документе

**Что сделать:** заменить описание, где "Мідли" выступают единственным наполнителем, на общий принцип: любой бакет может быть донором, средние — просто последние в очереди.

**Почему:** в коде любой бакет может отдавать игроков соседнему. Средние бакеты отдают чаще просто потому, что их посещают последними. Старое описание создавало ложное впечатление, что только "Мідли" могут быть донорами.

### Место 1 — секция "Group Creation Principle"

**Было:**
```
Пріорітетною для заповнення має бути група A "Нуби", а потім  група C "Топи". Група B "Мідли" збираються останні
```

**Стало:**
```
Бакети заповнюються від країв до центру: спочатку "Нуби", потім "Топи", "Мідли" — останні (детальніше див. **Group Creation Logic (for N groups)**)
```

### Место 2 — буллет про наповнювач

**Было:**
```
Якщо загальна кількість гравців дозволяє створити 3 окреми групи, але при цьому не вистачає гравців у брекетах A "Нуби" або\\та  C "Топи" , то в цьому випадку гравці з брекету B "Мідли" будуть виступати "наповнювачем" для інших бакетів, і відповідно частина гравців  брекету B "Мідли" має бути доєднана до іншого бакета у кількості достатній для створення окремої групи
```

**Стало:**
```
Якщо загальна кількість гравців дозволяє створити 3 окремі групи, але при цьому не вистачає гравців у брекетах A "Нуби" або/та C "Топи" — гравці підтягуються з сусідніх ще не відвіданих бакетів. Середні бакети відвідуються останніми, тому найчастіше саме вони виступають донорами — але будь-який бакет може віддати гравців сусідньому за потреби
```

## Заметки (не правки, а наблюдения)

### MinSize×2 проверка (шаги 2.1/2.2 в алгоритме)

GDD описывает: "если < MinSize×2 → одна группа". В коде такой явной глобальной проверки нет — это эмерджентное поведение (мало участников → всё в 1 бакет → 1 группа). **Результат совпадает**, но механизм другой. Можно оставить как есть — упрощение корректно как ментальная модель.

### Текст "завжди тільки три групи - A, B і С" (секция "Flow з точки зору гравця")

С MaxGroupCount > 3 или при использовании TargetSize/MaxSize групп может быть больше трёх. Сейчас это корректно для текущего деплоймента, но при расширении может стать неточным.

### Опечатки в английских названиях

В оригинале `"Midle"` / `"Midles"` — исправлено на `"Middle"` / `"Middles"` в примерах и JSON-блоках.
