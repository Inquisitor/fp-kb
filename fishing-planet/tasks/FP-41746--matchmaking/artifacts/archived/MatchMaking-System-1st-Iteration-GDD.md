# General

Документ описує першу ітерацію системи матчмейкінгу - спеціального алгоритму, який проводить "сортування" кворуму гравців, які зареєструвалися на участь в змаганнях та розділяє іх на групи (новачки - середній рівень - профі) для забезпечення більш комфортної гри та "справедливого" розподілу винагород.

**В першій ітерації нагороди для всіх груп будуть однаковими.**

В наступних ітераціях планується зробити винагороди різної цінності для окремих груп.

## Benefits for Developers

Найголовніше для нас  - це надати більшої кількості гравців можливість претендувати на нагороди та тим самим залучити більшу кількість гравців до змагальної активності

Таким чином ми плануємо підвищити наступні показники:

* **Retention Rate,** шляхом отримання додаткової винагороди та шансом здобути місце в пов'язаних лідербордах
* **Monetization** - не напряму, проте гравці які серйозно грають в змаганнях, як правило активніше донатять на екіпірування

## Benefits for Players

Для гравців нова система матчмейкінгу надає наступні переваги:

* Вищі шанси отримати винагороду у вигляді внутрішньоігрової валюти та спорядження за рахунок більш справедливого розподілу гравців у змаганнях
* Можливість для самовираження шляхом  порапляння в лідерборди

## Glossary

**Glossary**

**Bracket**
Розподіл гравців за рейтингом у межах визначених діапазонів _(min–max)_.
Діапазони брекетів не повинні перетинатися або мати прогалини.

Приклад: _(10 / 6 / 7)_ — кількість гравців у кожному брекеті.

---

**Bucket**
Початкове об'єднання гравців на основі мінімально допустимого розміру групи.
Якщо окремі брекети не досягають мінімального розміру, вони об'єднуються.

Приклад: загальна кількість гравців — 23, мінімальний розмір групи — 20 → _(23 / 0 / 0)_

---

**Group**
Фінальний результат матчмейкінгу.
Набір гравців, які беруть участь у змаганні та розподіляють між собою повний комплект винагород.

Приклад: _(23 / 0 / 0)_ — одна група з 23 гравців на момент старту.

# Feature Description

Матчмейкінг (система підбору оппонентів) - це фіча, **яка є складовою і невідємною частиною  змаганнь** . Системи формування кворуму для участі в турнірах та UGC (User Generated Competition) працюють за іншим принипом **і розглядатися не будуть.**

В основі матчмейкінгу лежить принцип розбиття зареєстрованого кворуму гравців на окремі групи ( у відповідності до значення їхнього змагального рейтингу).
**Якщо у гравця , який реєструється в змаганні не має рейтингу йому присвоюється нульовий рейтинг.**  Гравці в середині кожної групи будуть змагатися виключно між собою, і на кожну групу окремо буде формуватися своя таблиця результатів (**з первинним та вторинним скорінгом**) і видаватися свій пул винагород (**але складові цього пулу для всіх груп однакові в першій ітерації**).

## **Group Creation Principle ( example for 3 groups)**

 Відповідно базовою одиницею оперування в матчмейкінгу буде виступати **Група, Group:**

Пріорітетною для заповнення має бути група A "Нуби", а потім  група C "Топи". Група B "Мідли" збираються останні

* Група - це певна кількість гравців (від 20 і більше), які відсортовані по максимально близькому значенню _**personal сompetitive rating (PCR).**_
* **Основна ціль - віділити гравців з низько рівневого брекета "Нуби" в окремий Bucket таким чином максимально відокремити їх від інших гравців.**
* Групи формуються вже після того як реєстрацію на змагання завершено, коли буде точно відомо кількість та параметри всіх гравців в групі.
* Первинним параметром розбиття на **Bucket** буде кількість балів _**personal сompetitive rating**_ **(PCR),** в залежності від цього може бути 3 **Бакети**:

| personal competitive rating points брекету) | код внутрішні назви Бакету |
| --- | --- |
| від 0 до 100 (Bracket A) | "Нуби" / Newbies |
| від 101 до 1000 (Bracket B) | "Мідли" / Midle |
| від 1001+(Bracket C) | "Топи" / Top |

Бажаним сетапом кожного змагання буде 3 групи по 20+ гравців, по 1 на кожний брекет

Таким чином в JSON кожного змагання буде прописано ці налаштування, приклад (детальний опис в **Parameters in competition JSON**):

```json
  "Grouping" : {
    "MinSize" : 20,
    "Groups" : [
      { "GroupId" : 1, "GroupName" : "Newbies", "MinRating" : 0,  "RatingMultiplier" : 1.0, "RewardMultiplier" : 1.0 },
      { "GroupId" : 2, "GroupName" : "Midles", "MinRating" : 101, "RatingMultiplier" : 1.0, "RewardMultiplier" : 1.0 },
      { "GroupId" : 3, "GroupName" : "Tops", "MinRating" : 1001,  "RatingMultiplier" : 1.0, "RewardMultiplier" : 1.0 }
    ]},
```

Мінімально можлива кількість учасників змагання - **20 зареєстрованих** гравців.

* **Якщо на момент старту змагання зареєестровано менше ніж 20 гравців, то замагання не стартує з формулюванням "не вистачає учасників для старту змаганнь"** (вже реалізовано в поточній системі)
* **Якщо загальна кількість гравців дозволяє створити 3 окреми групи, але при цьому не вистачає гравців у брекетах A "Нуби" або\\та  C "Топи" , то в цьому випадку гравці з брекету B "Мідли" будуть виступати "наповнювачем" для інших бакетів, і відповідно частина гравців  брекету B "Мідли" має бути доєднана до іншого бакета у кількості достатній для створення окремої групи**
* **При можливості бажано розводити гравців з одного клубу та друзів в різні групи (в першій ітерації не реалізовано)**

## Group Creation Algorithm **(for 3 groups)**

**Два базових постулати які потрібно памятати при формуванні груп для проведення змагання:**

1. **Якщо змагання можливо провести ( є мінімальна кількість зареєстрованих людей) - ми його проводимо, навіть якщо немає можливості створити різні групи**
2. **При розподіленні гравців, якщо це можливо гравці з брекету Нуби мають бути відокремлені і грати виключно між собою - Мідли і Топи можуть бути разом**

1. Перевіряємо чи кількість зареєстрованих не менше 20-ти на момент старту змагання:

1.1 Якщо **менше 20 гравців - замагання не стартує** з формулюванням "не вистачає учасників для старту змаганнь"

1.2 Якщо є 20 і більше зареєстрованих гравців - змагання стартує

2. Дивимось кількість гравців які зареєстровані в змаганні і визначаємо чи можна розбити на групи (дивимось на параметр `"MinSize"` в json змагання):

2.1 Якщо кількість гравців **менше** `"MinSize"`\*2 - проводимо змагання в рамках **однієї групи - не робимо розбивку**

2.2 Якщо кількість учасників **дорівнює** `"MinSize"`\*2 чи більше - **можна формувати групи**

3. Далі, якщо можна формувати групи, дивимось значення рейтингу PCR гравців та формуємо початкові бакети згідно розподілення гравців по брекетам
4. Оскільки у нас з початку є 3 основні брекети, ми формуємо для цих гравців 3 відповідні бакети. Всі гравці початково розташовуються по бакетам згідно їх місць по брекетам (ми розташовуємо  в одному бакеті нубів, в іншому бакеті - мідлів , і в останньому бакеті топів).
5. Після цього ми балансуємо гравців між бакетами. Якщо в  бакеті мінімум 20 гравців - то бакет вважається вже сформованим
6. **Стартуємо змагання і попередньо сформовані бакети перетворюються на групи в момент старту змагання**
7. Після змагання

7.1 Видаємо винагороду на кожну сформовану групу (10 призових місць). **В результатах відображаємо всі групи у вигляді окремої таблиці на кожну групу**

7.1 Нараховуємо бонусний рейтинг переможцям та пенальті рейтинг для програвщих

**Публічно ми показуємо гравцям тільки назви груп по латинському алфавіту "A", "B", "C"..."G"…. "T"… тощо**

## **Group Creation Logic (for N groups)**

В цілому  логіка заповнення для  "n" групп аналогічна логіці заповнення 3 груп:
1\. Ми формуємо n брекетів з діапазонами рейтингу

2 . З брекетів них формуються бакети - п**ріорітетним для заповнення має бути бакет  "Нуби", а потім   -"Топи".**
3\.  Далі ми послідовно заповнюємо n бакетів, при при заповненні ми спочатку заповнюємо **бакет "мідлів"** з найменшим рейтингом, далі (при формуванні наступного бакета) заповнюємо бакет мідлів з найбільшим рейтингом), потім знову з найменшим..

## Logic for creating subgroups

В попередніх розділах наголошується що мета нового мачмейкінгу розділити в змаганні гравців з низьким рейтингом від гравців з високим рейтингом. Це основна мета. Також пояснюється що для цього достатньо трьох груп.

Тому в json всіх змагань прописані тільки 3 групи з чітко вказаними діапазонами рейтингу. Щоб гравці  з цих диапазонів не перетинались у змаганні.

Тобто якщо в змагання зареєструвалось 300 гравців з діапазону 1-100 рейтингових балів то вони всі попадуть в одну групу (за умови що сформовано вже 2 і 3 группу з іншими діапазонами). Це так і передбачалося.

Але якщо знадобиться зробити розділення на більшу кількість груп, то можна використати в json змігання параметри **TargetSize**,

TargetSize - це цільова кількість гравців в підгрупі. Цей параметр діє на всі три групи.

Тобто, наприклад, якщо ми поставимо **TargetSize**=29 при **MinSize**=15, то якщо в змаганні зареєструються 100 гравців нубів, 100 гравців мідлів, 100 гравців топів, то отримаємо розбиття на 12 підгруп. Тобто в кожній групі буде прибизно 25 гравців. Назви підгруп будуть від A до Z.

<!-- Image: TargetSize subgroup splitting diagram -->

Треба зауважити що **TargetSize** може бути тільки менше чи дорівнювати (**MinSize**\*2-1). Але це не значить що якщо **TargetSize**=29 то кількість в підгрупі буде завжди менша чи дорівнювати 29.

## Use Cases (For 3 groups)

**Приклад 1**: `Умова: "MinSize" : 15, "TargetSize" немає.` В змаганні зареєструвалася 81 людина з них потенційно 41 - група "Нуби", група "Мідли" -30, група "Топи" - 10 .

1. Перевіряємо чи людей рівно чи більше `"MinSize"*2` . 81 більше ніж 30 - значить потенційно робимо більше однієї групи
2. Формуємо групу "Нуби". 41 більше 15, залишок 81-41=40, що достатнью щоб сформувати групу "Нуби" та ще одну групу як мінімум
3. Формуємо групу "Топи". у нас є 10 потенційних гравців групи "Топи" , що недостатньо для формування групи. Додаємо 5 гравців з залишку з найвищим рейтингом PCR і таким чином отримуємо групу "Топи" в складі 15 осіб
4. Формуємо групу "Мідли". Після формування груп "Нуби" та "Топи" залишок 61-21-15=25 гравців, що більше 15 та достатньо щоб сформувати групу "Мідли" зі складом в 25 людей.
5. Видаємо 10 призових місць винагороди групі "Нуби" (41 гравець відповідно) , 10 призових місць винагороди групі "Топи" (15 гравців), та 10 призових місць групі "Мідли" (25 гравців відповідно)

**Приклад 2**: `Умова: "MinSize" : 15, "TargetSize" немає.` В змаганні зареєструвалася 61 людина з них 55 - група "Топи", Група "Мідли" -3, група "Нуби" - 3 .

1. Перевіряємо чи людей рівно чи більше `"MinSize"*2` . 61 більше ніж 30 - значить потенційно робимо більше однієї групи
2. Формуємо групу "Нуби". 61 більше 15, але залишок 61-55=6, в такому випадку ми додаємо до групи "Нуби" гравців з інших груп з найнижчим рейтингом у кількості достатній щоб в групі "Нуби" було 15 гравців = 61 - (6 +3+6). Таким чином у групу "Нуби" переходить 3 гравця з групи "Мідли" та 6 гравців групи "Топи" (останні з найменшим значенням рейтингу з поміж "топів")
3. Формуємо групу "Топи" - 61-15 = 46, що більше 15 і значить ми формуємо групу "Топи" у кількості 46 гравців
4. Формуємо групу "Мідли". Оскільки потенційні гравці цієї групи були використані для наповнення групи "Нуби" у повному складі, то формування цієї не відбувається
5. Оскільи группу мідлів ми не сформовали, а группа топів має 46 гравців - додатково ділимо группу топи на 2 группи по 23 гравця
6. Видаємо 10 призових місць  групі "Нуби" (15 гравців відповідно) , та 10 призових місць винагороди 2 групам "Топи" (23 і 23 гравця відповідно)

**Приклад 3**: `Умова: "MinSize" : 15, "TargetSize" немає.` В змаганні зареєструвалося 99 людей з них група "Нуби" -1 , Група "Мідли" -1, група "Топи" - 97

1. Перевіряємо чи людей рівно чи більше `"MinSize"*2` . 99 більше ніж 30 - значить потенційно робимо більше однієї групи
2. Формуємо групу "Нуби". 1 менше 15, залишок 99-1=98, що недостатнью щоб сформувати групу "Нуби" з потенційних гравців групи "Нуби" але залишок дозволяє створити группу "Нуби" та ще одну додаткову групу відповідно . Формуємо групу "Нуби" з 1 гравця + 14 гравців з залишку з найменшим рейтингом PCR
3. Формуємо групу "Топи". 97 -більше 15, залишок 99-15 =84 , що достатньо для формування групи. Таким чином отримуємо групу "Топи" в складі 84 осіб
4. Формуємо групу "Мідли". З попередніх кроків виходить що 1 потенційний гравець з групи "Мідли" був використаний для "наповнення" групи "Нуби" і виходить що формувати групу немає з кого
5. Оскільки у нас сформовано 2 группи а їх має бути 3, то ділимо группу "Топи у складі" 84 осіб на 2 группи по 42 особи в кожній
6. Видаємо 10 призових місць винагороди групі "Нуби" (15 гравців відповідно) , 10 призових місць винагороди 2 групам "Топи" (42 гравців 2 групи), групу "Мідли" ми не сформували

**Приклад 4**: `Умова: "MinSize" : 15, "TargetSize" немає.` В змаганні зареєструвалася людина з них потенційно 14 - група "Нуби", група "Мідли" -14, група "Топи" - 14.

1. Перевіряємо чи людей рівно чи більше `"MinSize"*2` . (14+14+14) = 42 що більше ніж 30 -  значить потенційно робимо більше однієї групи
2. Формуємо групу "Нуби". 14 менше 15, залишок 42-14= 28, що недостатнью щоб сформувати групу "Нуби" з потенційних гравців групи "Нуби" але залишок дозволяє створити группу "Нуби" та ще одну додаткову групу відповідно .
3. Формуємо групу "Топи". Залишок 28 менше 29 що , що недостатньо для формування ще однієї групи. Таким чином отримуємо групу "Топи" в складі 28 гравців (14 потенційних гравців групи "Топи" + 14 гравців залишку, яких недостаньо щоб сформувати окрему групу)
4. Видаємо 10 призових місць винагороди групі "Нуби" (15 гравців відповідно) , 10 призових місць винагороди групі "Топи" (28 гравців), групу "Мідли" ми не сформували

## **MatchMaking System integration with Clubs**

В першій ітерації система матчмейкінгу яка використовується для змаганнь майже жодним чином не взаємодіє з клубами та лігами. Єдине виключення - винагорода в клубних баллах (Сlub Points) яка вручну прописується в джейсоні змагання за призові місця. **І відповідно потрібно буде зробити щоб дана винагорода (10 призових місць) звстосовувалася для 10 призових місць кожної групи, які утворилися в результаті розподілу кворуму змагання.**

В **майбутніх ітераціях** можлива більш тісна взаємодія системи матчмейкінгу з клубами **в рамках ексклюзивних клубних змаганнь** , які можуть в тому чи іншому вигляді використовувати матчмейкінг

# **Environment Variables**

**IsRatingByPlaceEnabled** - змінна для активації нової системи рейтингу і мачмейкінгу.

Якщо **IsRatingByPlaceEnabled**=Y, то значення рейтингу і налаштування груп береться з jsonу змагання. Якщо в змаганні не прописані бали рейтингу і групи мачмейкінгу, то ніякого рейтингу не нараховується і розбиття на групи не відбувається.

# Global **Variables**

Додано глобальну змінну **Tournaments.ParticipantGridMode = AA**, значення якої складається із 2х символів (**A або B**)

* перша літера означає режим гріда учасників під час змагання

    * A - показуємо УСІ групи і УСІХ учасників в групах
    * B - показуємо УСІ групи, але ТОП10 учасників в групі + мій запис + 3 навколо мене

* друга літера означає режим гріда учасників після завершення змагання

    * A - показуємо УСІ групи і УСІХ учасників в групах
    * B - показуємо УСІ групи, але ТОП10 перших місць (або топ WinnersCountAdmited2NextStage)
