---
name: ConfigJson schema extension requires a back-fill script
description: Any PR that extends the schema of a `ConfigJson` column populated into pre-generated DB rows must ship an idempotent back-fill SQL script in `SQL/Releases/`
type: feedback
---
Any PR that extends the schema persisted into a `ConfigJson` column for entities that are pre-generated into the DB ahead of time (e.g. `Tournaments.ConfigJson`, `TournamentTemplates.ConfigJson`) must ship a per-release idempotent fix-up SQL script in `<project>/SQL/Releases/` alongside the schema change.

**Why:** `ConfigJson` is a JSON snapshot of the entity at generation time. After a schema extension deploys, rows generated by the previous binary still carry the pre-extension shape — new fields are absent. Standard admin actions cannot rescue them: `Regenerate*` actions exclude rows whose registration/start window is already open, and the more aggressive variants cascade-delete participant registrations. Without a back-fill script the only options on deploy day are losing user state or hand-patching in prod.

**How to apply:**
1. Identify whether the entity is pre-generated with a future-facing window (e.g. `Tournaments` rows scheduled days ahead). If yes, this rule applies.
2. Ship a fix-up SQL script in `<project>/SQL/Releases/R<YYYYMM>-<topic>-FixUp.sql`. Requirements:
   - Idempotent — safe to re-run, safe on platforms where no rows match.
   - `@DryRun` switch — defaults to `1`; rolls back the wrapping transaction when `1`, commits when `0`.
   - Two phases minimum: (a) detect & report rows needing patch, plus any source-content defects that would block the patch; (b) apply patch.
   - Use `JSON_MODIFY` for field-level patches; copy from the source `ConfigJson` (e.g. the row's template) where possible.
   - For derived fields — those filled by deserialization-side initializers rather than authored content — compute the derived value directly in SQL, or land a code-side defensive fix first (preferred) so the back-fill only needs to copy the raw source.
3. The Server Release Checklist template carries a conditional step pointing at the per-release script, placed immediately after `Regenerate future competitive activities`. Verify that step is present in the per-release checklist copied from the template; if removed by mistake, restore it.

Worked example: `<project>/SQL/Releases/R202604-Leaderboards-Rating-Matchmaking-FixUp.sql` — three-phase script with `@DryRun` switch, ships with source-content audit and patch-cohort preview before commit.
